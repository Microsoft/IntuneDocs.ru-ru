## <a name="week-of-november-26-2018"></a>Неделя с 26 ноября 2018 г.

### <a name="app-management"></a>Управление приложениями

#### <a name="uninstalling-apps-on-corporate-owned-supervised-ios-devices----1281677---"></a>Удаление приложения на корпоративных защищенных устройствах iOS <!-- 1281677 -->

C корпоративных защищенных устройств iOS можно удалить любое приложение. Можно удалить любое приложение, выбрав для групп пользователей или устройств тип назначения **Удаление**. С личных или незащищенных устройств iOS можно удалить только те приложения, которые были установлены с помощью Intune.

#### <a name="downloading-intune-win32-app-content----2617320---"></a>Скачивание содержимого приложения Intune Win32 <!-- 2617320 -->
Клиенты Windows 10 версии RS3 и выше будут скачивать содержимое приложения Intune Win32 с помощью компонента оптимизации доставки на клиенте Windows 10. Оптимизация доставки включает возможности одноранговой доставки, включенные по умолчанию. Оптимизацию доставки сейчас можно настроить с помощью групповой политики, а в будущем — с помощью Intune MDM. См. дополнительные сведения об [оптимизации доставки для Windows 10](https://docs.microsoft.com/windows/deployment/update/waas-delivery-optimization). 

#### <a name="end-user-device-and-app-content-menu----2771453---"></a>Контекстное меню для устройства пользователя и приложения <!-- 2771453 -->
Теперь пользователи могут использовать контекстное меню на устройстве и в приложениях для активации распространенных действий, например переименования устройства и проверки соответствия требованиям.

#### <a name="set-custom-background-in-managed-home-screen-app-----3041945---"></a>Настройка пользовательского фона в приложении управляемого начального экрана <!-- 3041945 -->
Мы добавим параметр, который позволит настраивать внешний вид фона приложения управляемого начального экрана в Android для бизнеса на устройствах в режиме киоска с несколькими приложениями.  Чтобы настроить **Пользовательский URL-адрес фона**, перейдите в Intune на портале Azure > Конфигурация устройства. Выберите текущий профиль конфигурации устройства или создайте новый, чтобы изменить параметры киоска.
Дополнительные сведения о параметрах киоска см. в статье [Параметры ограничений для рабочих устройств в Intune](device-restrictions-android-for-work.md).

#### <a name="app-protection-policy-assignment-save-and-apply----3104570---"></a>Назначение политики защиты приложений — сохранение и применение <!-- 3104570 -->
Теперь у вас есть больше возможностей управления [назначениями политики защиты приложений](app-protection-policies.md#deploy-a-policy-to-users). Выбирая *Назначения*, чтобы задать или изменить назначения политики, необходимо **сохранить** конфигурацию, чтобы применить изменение. Чтобы очистить все внесенные изменения, не сохраняя их в списках включений или исключений, щелкните **Отменить**.  С помощью операций сохранения и отмены можно назначить политику защиты приложений только нужным пользователям.

#### <a name="new-microsoft-edge-browser-settings-for-windows-10-and-later----3174639---"></a>Новые параметры браузера Microsoft Edge для Windows 10 и более поздней версии <!-- 3174639 -->
В это обновление входят новые параметры для управления браузером Microsoft Edge на устройствах. Список этих параметров см. в статье [Параметры ограничений устройств Windows 10 (и более поздних версий) в Intune](device-restrictions-windows-10.md#microsoft-edge-browser).

#### <a name="new-apps-support-with-app-protection-policies----3330037---"></a>Поддержка новых приложений в политиках защиты приложений <!-- 3330037 -->
Теперь вы можете управлять следующими приложениями с помощью [политик защиты приложений Intune](app-protection-policies.md):
- Stream (iOS);
- To-Do (Android, iOS);
- PowerApps (Android, iOS);
- Flow (Android, iOS).

С помощью политик защиты приложений можно защитить данные организации и управлять передачей данных в этих приложениях, так же как и в других приложениях, управляемых политикой Intune. Примечание. Если приложение Flow не отображается в консоли, вы можете добавить его при создании или редактировании политики защиты приложений. Чтобы сделать это, выберите **+Другие приложения**, а затем укажите *идентификатор приложения* для Flow в поле ввода. Для Android укажите *com.microsoft.flow*, а для iOS — *com.microsoft.procsimo*.

#### <a name="intune-app-protection-policies-ui-update----3251427---"></a>Обновление пользовательского интерфейса политики защиты приложений Intune <!-- 3251427 -->
Мы изменили подписи параметров и кнопок для защиты приложений Intune, сделав их более понятными. Вот некоторые изменения:  

- Элементы управления изменены с **да** / **нет** в основном на элементы управления **блокировать** / **разрешить** и **отключить** / **включить**. Подписи к ним также обновлены.  
- Формат параметров изменен так, чтобы параметр и его подпись располагались рядом на элементе управления для более удобной навигации.   

Параметры по умолчанию и число параметров остаются прежними, но с этим изменением станет проще и удобнее искать и использовать параметры, понимать их предназначение и применять выбранные политики защиты приложения.

### <a name="device-configuration"></a>Конфигурация устройства

#### <a name="ios-and-macos-version-numbers-and-build-numbers-are-shown----1892471---"></a>Отображаются номера версии и номера сборки iOS и macOS <!-- 1892471 -->.
В разделе **Соответствие устройства политике** > **Соответствие устройства политике** отображаются версии операционных систем iOS и macOS, которые доступны для использования в политиках соответствия требованиям. В этом обновлении можно настроить номер сборки для обеих платформ.
При выпуске обновлений безопасности Apple, как правило, оставляет номер версии прежним, но меняет номер сборки. Используя номер сборки в политике соответствия, можно легко проверить, установлено ли обновление для защиты от уязвимостей.
Чтобы использовать эту функцию, см. политики соответствия требованиям для [iOS](compliance-policy-create-ios.md#device-health) и [macOS](compliance-policy-create-mac-os.md#device-properties).

#### <a name="update-rings-are-being-replaced-with-delivery-optimization-settings-for-windows-10-and-later----2753807---"></a>Круги обновления заменяются на параметры оптимизации доставки для Windows 10 и более поздних версий <!-- 2753807 -->
Оптимизация доставки — это новый профиль конфигурации в Windows 10 и более поздних версиях. Эта функция предоставляет упрощенную доставку обновлений программного обеспечения на устройства в организации. Это обновление также поможет вам предоставлять параметры в новых и существующих кругах обновления с помощью профиля конфигурации.
Чтобы настроить профиль конфигурации оптимизации доставки, см. статью, посвященную [параметрам оптимизации доставки для Windows 10 и более поздних версий](delivery-optimization-windows.md).

#### <a name="temporarily-pause-kiosk-mode-on-android-devices-to-make-changes----3041935---"></a>Временная приостановка режима киоска на устройствах Android для внесения изменений <!-- 3041935 -->
При использовании устройств Android в режиме киоска с несколькими приложениями ИТ-администратору может потребоваться внести изменения на устройстве. В это обновление входят новые параметры киоска с несколькими приложениями, которые позволяют ИТ-администратору временно приостановить режим киоска с помощью PIN-кода и получить доступ ко всему устройству.
Дополнительные сведения о параметрах киоска см. в статье [Параметры ограничений для рабочих устройств в Intune](device-restrictions-android-for-work.md).

#### <a name="enable-virtual-home-button-on-android-enterprise-kiosk-devices-----3042021---"></a>Включение виртуальной кнопки домашней страницы на устройствах киоска Android для бизнеса <!-- 3042021 -->
Новый параметр позволит пользователям с помощью программной клавиши на устройстве переключаться между приложением управляемого начального экрана и другими назначенными приложениями в киоске с несколькими приложениями. Вы сможете настроить этот параметр для корпоративных устройств Android единого назначения. Чтобы включить или отключить **виртуальную кнопку домашней страницы**, перейдите в Intune на портале Azure > конфигурация устройства. Выберите текущий профиль конфигурации устройства или создайте новый, чтобы изменить параметры киоска.
Дополнительные сведения о параметрах киоска см. в статье [Параметры ограничений для рабочих устройств в Intune](device-restrictions-android-for-work.md).

### <a name="support-for-ios-12-oauth-in-ios-email-profiles---2155106---"></a>Поддержка iOS 12 OAuth в профилях электронной почты iOS <!--2155106 -->

Профили электронной почты iOS в Intune поддерживают Open Authorization (OAuth) iOS 12. Чтобы увидеть эту функцию в действии, создайте новый профиль (**Конфигурация устройства** > **Профили** > **Создать профиль** > **iOS** > **Электронная почта** в качестве типа профиля) или измените существующий профиль электронной почты iOS. При включении OAuth в профиле, который уже развернут для пользователей, пользователям предлагается повторно выполнить проверку подлинности и скачать свою электронную почту.

Статья [Профили электронной почты iOS](email-settings-ios.md) содержит дополнительные сведения об использовании OAuth в профиле электронной почты.

Дополнительные сведения см. в статье, посвященной [развертыванию гибридных устройств, присоединенных к Azure AD, с помощью Intune и Windows Autopilot](windows-autopilot-hybrid.md).
Эта функция станет доступна для пользователей через несколько дней. Вы, возможно, не сможете выполнить эту процедуру, пока функция не будет развернута в вашей учетной записи.

## <a name="week-of-october-29-2018"></a>Неделя начиная с 29 октября 2018 г.

### <a name="app-management"></a>Управление приложениями

#### <a name="require-non-biometric-pin-after-a-specified-timeout----1506985---"></a>Запрашивание небиометрического ПИН-кода после указанного времени ожидания <!-- 1506985 -->
Благодаря требованию предоставлять небиометрический ПИН-код по истечении времени ожидания, заданного администратором, Intune повышает безопасность приложений с поддержкой управления мобильными приложениями (MAM), ограничивая доступ к корпоративным данным с помощью биометрической идентификации. Эти параметры касаются тех пользователей, которые применяют Touch ID (iOS), Face ID (iOS), Android Biometric или другие способы биометрической проверки подлинности, которые могут появиться в будущем, для доступа к приложениям с поддержкой политики защиты приложений и MAM. Они позволят администраторам Intune более детально контролировать доступ пользователей и избежать случаев, когда не имеющий на то права пользователь может получить доступ к корпоративным данным с помощью устройства, на котором настроено несколько биометрических методов доступа, например по отпечатку пальца. На портале Azure откройте **Microsoft Intune**. Последовательно выберите **Клиентские приложения** > **Политики защиты приложений** > **Добавить политику** > **Параметры**. Найдите раздел **Доступ**. Сведения о параметрах доступа см. в разделах, посвященных [параметрам iOS](app-protection-policy-settings-ios.md#access-settings) и [параметрам Android](app-protection-policy-settings-android.md#access-settings).

#### <a name="intune-app-data-transfer-settings-on-ios-mdm-enrolled-devices----2244713---"></a>Параметры передачи данных в приложении Intune на зарегистрированных в MDM устройствах iOS <!-- 2244713 -->
Вы можете разделить возможность управления параметрами передачи данных в приложении Intune на зарегистрированных в MDM устройствах iOS и определение удостоверения зарегистрированного пользователя (также называемого именем участника-пользователя). Администраторы, которые не используют конфигурацию IntuneMAMUPN, не увидят изменений в поведении. Когда эта функция станет доступна, администраторы, использующие IntuneMAMUPN для контроля передачи данных на зарегистрированных устройствах, должны будут ознакомиться с новыми параметрами и при необходимости обновить конфигурацию своих приложений.

#### <a name="windows-10-win32-apps----2617325---"></a>Приложения Win32 Windows 10 <!-- 2617325 -->
Вы можете настроить приложения Win32 для установки в контексте пользователя, чтобы установка выполнялась для отдельных, а не для всех пользователей устройства.

### <a name="intune-app-protection-policies----2984657---"></a>Политики защиты приложений Intune <!-- 2984657 -->
Политики защиты приложений Intune позволяют настроить различные параметры защиты данных для приложений, защищаемых Intune, таких как Microsoft Outlook и Microsoft Word. Мы изменили оформление этих параметров для [iOS](app-protection-policy-settings-ios.md) и [Android](app-protection-policy-settings-android.md), чтобы упростить поиск отдельных параметров. Существует три категории параметров политик.
- **Перемещение данных** — эта группа включает элементы управления защиты от потери данных (DLP), включая ограничения вырезания, копирования, вставки и сохранения в другом формате. Эти параметры определяют, как пользователи взаимодействуют с данными в приложениях.
- **Требования доступа** — эта группа содержит параметры ПИН-кода на уровне приложения, которые определяют, каким образом конечный пользователь получает доступ к приложениям в рабочем контексте.  
- **Условный запуск** — эта группа содержит такие параметры, как минимальные настройки операционной системы, обнаружение взломанных устройств и устройств с административным доступом, а также автономные льготные периоды.  
  
Функциональные возможности этих параметров не изменились, однако их стало проще найти при работе над политикой.

#### <a name="issue-scep-certificates-to-user-less-devices----1744554---"></a>Выдача сертификатов SCEP устройствам без пользователя <!-- 1744554 -->
Сейчас сертификаты выдаются пользователям. В этом обновлении можно выдавать сертификаты SCEP устройствам, в том числе устройствам без пользователя, таким как киоски (последовательно выберите **Конфигурация устройства** > **Профили** > **Создать профиль** > **платформа Windows 10 и более поздние версии** > **Сертификат SCEP**). Доступны другие обновления, в том числе:
- Свойство **Субъект** в профиле SCEP теперь является пользовательским текстовым полем и может включать новые переменные. 
- Свойство **Альтернативное имя субъекта (SAN)** в профиле SCEP теперь отображается в формате таблицы и может включать новые переменные. В этой таблице администратор может добавить атрибут и добавить значение в пользовательское текстовое поле. SAN поддерживает следующие атрибуты: 
  - DNS
  - Адрес электронной почты
  - имя участника-пользователя

  Эти новые переменные можно добавить как статический текст в пользовательское текстовое поле. Например, атрибут DNS можно добавить в виде `DNS = {{AzureADDeviceId}}.domain.com`.

  > [!NOTE]
  > В статическом тексте SAN не используются фигурные скобки, точка с запятой и вертикальная черта ({ } ; |). Фигурные скобки могут содержать только одну из новых переменных сертификата устройства. Только тогда их можно использовать с `Subject` или `Subject alternative name`. 

Новые переменные сертификата устройства:  

```
"{{AAD_Device_ID}}",
"{{Device_Serial}}",
"{{Device_IMEI}}",
"{{SerialNumber}}",
"{{IMEINumber}}",
"{{AzureADDeviceId}}",
"{{WiFiMacAddress}}",
"{{IMEI}}",
"{{DeviceName}}",
"{{FullyQualifiedDomainName}}",
"{{MEID}}",
```

> [!NOTE]
>  - `{{FullyQualifiedDomainName}}` работает только на устройствах Windows и устройствах, присоединенных к домену. 
>  -  При указании свойств устройства, таких как номер IMEI, серийный номер и полное доменное имя, в субъекте или SAN для сертификата устройства, имейте в виду, что эти свойства могут быть подделаны пользователем, имеющим доступ к устройству. 

В разделе [Создание профиля сертификата SCEP](certificates-scep-configure.md#create-a-scep-certificate-profile) перечислены текущие переменные для создания конфигурируемого профиля SCEP.

ИТ-администратор Intune может принудительно применить определенный список производителей устройств с Android и (или) моделей iOS, используя политики защиты приложений Intune. Политики защиты приложений Intune применяются только к устройствам с iOS и Android. С таким списком можно выполнить два отдельных действия:
- блокировать доступ из приложений на устройствах, не указанных в таком списке;
- выборочно очистить корпоративные данные на устройствах, не указанных в таком списке. 

Пользователь не сможет получить доступ к целевому приложению, если не соблюдены требования политики. В зависимости от параметров, либо пользователь может быть заблокирован, либо корпоративные данные могу быть выборочно очищены в приложении. На пользовательских устройствах клиент Intune будет выполнять действия на основе простого сопоставления строк, указанных в колонке Intune для политик защиты приложения. Этого можно сделать, используя список различных изготовителей устройств и моделей устройств, назначенных небольшой группе пользователей. В Microsoft Intune выберите **Клиентские приложения** > **Политики защиты приложений**, чтобы просмотреть и добавить политики защиты приложения. Дополнительные сведения о политиках защиты приложений см. в разделах [Что такое политики защиты приложений](app-protection-policy.md) и [Выборочная очистка данных с помощью действий доступа политики защиты приложений в Intune](app-protection-policies-access-actions.md).

### <a name="use-fips-mode-with-the-ndes-certificate-connector----1333688---"></a>Использование режима FIPS вместе с соединителем сертификатов NDES <!-- 1333688 -->
При установке соединителя сертификатов NDES на компьютере с включенным режимом федерального стандарта обработки информации (FIPS) выдача и отмена сертификатов работает неправильно. В этом обновлении поддержка TLS 1.2 включается в состав соединителя сертификатов NDES. 

Это обновление также содержит следующее:

- Соединителю сертификатов NDES требуется платформа .NET 4.5 Framework, автоматически входящая в состав Windows Server 2016 и Windows Server 2012 R2. Ранее .NET 3.5 Framework являлась минимальной требуемой версией.
- Поддержка TLS 1.2 входит в состав соединителя сертификатов NDES. Если сервер с установленным соединителем сертификатов NDES поддерживает TLS 1.2, то используется TLS 1.2. Если сервер не поддерживает TLS 1.2, то используется TLS 1.1. Сейчас TLS 1.1 используется для проверки подлинности между устройствами и сервером.

Дополнительные сведения см. в разделе [Настройка и использование сертификатов SCEP](certificates-scep-configure.md) и [Настройка и использование сертификатов PKCS](certficates-pfx-configure.md).

#### <a name="new-windows-defender-credential-guard-settings-added-to-endpoint-protection-settings---1102252-----from-1802-and-1804--"></a>Добавлены новые параметры Credential Guard в Защитнике Windows в раздел параметров Endpoint Protection <!--1102252 --><!--from 1802 and 1804-->

В этом обновлении раздел [Credential Guard в Защитнике Windows](https://docs.microsoft.com/windows/access-protection/credential-guard/credential-guard) (**Конфигурация устройств** > **Профили** > **Endpoint Protection**) включает следующие параметры. 

- **Credential Guard в Защитнике Windows**: активирует Credential Guard с функциями безопасности на основе виртуализации. Эта возможность позволяет защитить учетные данные во время следующей перезагрузки, если включены функции **Уровень безопасности платформы с безопасной загрузкой** и **Безопасность на основе виртуализации**. Возможны следующие значения.
  - **Отключено**: удаленно отключает компонент Credential Guard, если он ранее был включен с параметром **Включено без блокировки**.

  - **Включено с блокировкой UEFI**: блокирует отключение компонента Credential Guard с помощью раздела реестра или групповой политики. Чтобы отключить Credential Guard после использования этого параметра, необходимо отключить групповую политику. Затем следует удалить функции безопасности с каждого компьютера в непосредственном присутствии пользователя. Эти действия очистят конфигурацию, сохраненную в UEFI. Пока сохраняется конфигурация UEFI, компонент Credential Guard включен.

  - **Включено без блокировки**: позволяет удаленно отключать компонент Credential Guard с помощью групповой политики. На устройствах, на которых используется это значение, должна быть установлена по меньшей мере Windows 10 (версии 1511).

При настройке Credential Guard автоматически включаются следующие зависимые технологии. 

  - **Включение безопасности на основе виртуализации (VBS)**: включает функции безопасности на основе виртуализации (VBS) при следующей перезагрузке. Функции безопасности на основе виртуализации используют гипервизор Windows для поддержки служб безопасности и требуют безопасной загрузки.
  - **Безопасная загрузка с прямым доступом к памяти (DMA)**: включает VBS с безопасной загрузкой и прямым доступом к памяти. Средства защиты DMA требуют аппаратной поддержки и включаются только на корректно настроенных устройствах.

## <a name="notices"></a>Объявления

### <a name="plan-for-change-exchange-online-to-intune-connector-will-not-be-available-in-intune----3105122---"></a>Планирование изменений. Соединитель Exchange Online с Intune не будет доступен в Intune <!-- 3105122 -->
Для упрощения работы с Exchange Online и условным доступом мы отключим соединитель служб Exchange Online и Intune. Реализация этого изменения начнется в декабрьском обновлении службы и завершится в обновлении службы за февраль 2019 г.

#### <a name="how-does-this-affect-me"></a>Как это повлияет на работу?
Вы получили это сообщение, так как, по нашим сведениям, вы можете использовать функцию соединителя служб в своей среде. Соединитель служб поддерживает управление с помощью Intune для устройств Exchange Active Sync Only в Exchange Online и не поддерживает локальную инфраструктуру. Этот соединитель отображается в консоли таким образом, как будто он необходим для условного доступа, но на самом деле это не так. Чтобы внести ясность, в декабрьском обновлении службы Intune в консоли будет отключена кнопка для настройки новых соединителей. Затем в феврале 2019 г. все существующие соединители служб Exchange Online и Intune будут отключены.

Если вы используете эти соединители в своей среде, после отключения соединителей в феврале вы не сможете отслеживать или очищать устройства Exchange Active Sync Only в Intune. Это изменение не должно повлиять на пользователей.

#### <a name="what-can-i-do-to-prepare-for-this-change"></a>Как можно подготовиться к этому изменению?

Если вы настроили соединитель служб и используете устройства Exchange Active Sync Only, перейдите на другие методы управления устройствами. Можно выбрать один из следующих вариантов.

- зарегистрировать устройства в решении "Управление мобильными устройствами" (MDM);
- использовать политики защиты приложений Intune для управления устройствами;
- использовать элементы управления Exchange, как описано в этой документации.

### <a name="plan-for-change-performance-updates-to-intune-for-education---1750215--"></a>Планирование изменений: обновления для повышения производительности в Intune для образовательных учреждений <!--1750215-->
Мы добавили некоторые обновления в Intune для образовательных учреждений, которые позволят увеличить быстродействие и надежность при назначении параметров пользователям или устройствам. В рамках этого изменения к концу ноября мы перенесем ваши назначения политик или параметров в новые группы.

#### <a name="how-does-this-affect-me"></a>Как это повлияет на работу?

Как клиент Intune для образовательных учреждений, вы располагаете двумя динамическими группами Azure Active Directory (Azure AD): "Все пользователи" и "Все устройства". В рамках этих обновлений группы "Все пользователи" и "Все устройства" Azure AD перестанут отображаться в консоли Intune для образовательных учреждений. При этом они по-прежнему будут отображаться в консоли Intune в Azure и будут переименованы в "Все пользователи (устарело, не использовать)" и "Все устройства (устарело, не использовать)".

После применения обновлений вам больше не потребуется использовать группы Azure AD для назначения приложений и параметров в Intune. Мы переместим назначения параметров в новые группы в консоли Intune для образовательных учреждений, которые, как и прежде, будут называться "Все пользователи" и "Все устройства". Эти изменения реализуются в серверной части, поэтому вы не заметите никаких изменений в консоли Intune для образовательных учреждений. Влияния на конечных пользователей или зарегистрированные устройства не ожидается. 

#### <a name="what-do-i-need-to-do-to-prepare-for-this-change"></a>Как можно подготовиться к этому изменению?
Во время перемещения назначений политики никакие действия с вашей стороны не требуются. Если сейчас вы назначаете политики в консоли Intune для образовательных учреждений, продолжайте это делать.

Если сейчас вы назначаете политики группам Azure AD, упомянутым выше, в Intune в Azure, назначьте их группе "Все пользователи" или "Все устройства" в консоли Intune для образовательных учреждений. Как только вы увидите, что группы Azure AD переименованы в устаревшие в консоли, прекратите назначение политик в Azure AD. Если вы сейчас не используете переименованные группы для других целей, их следует удалить.

- Попросите своих пользователей обновить устройства на поддерживаемую версию ОС до декабря 2018 г. 
- Просмотрите отчеты Intune в консоли Intune в Azure, чтобы узнать, на какие устройства или пользователей может повлиять это изменение. Перейдите в раздел "Устройства" > "Все устройства" и выполните фильтрацию по ОС. Вы можете добавить дополнительные столбцы, чтобы узнать, кто в вашей организации пользуется устройствами с macOS 10.11. 
- Если вы используете гибридное управление мобильными устройствами (MDM), в консоли Configuration Manager последовательно выберите "Активы и соответствие" > "Устройства", щелкните правой кнопкой мыши столбцы для добавления столбцов операционной система и версии клиента, а затем выполните сортировку по ОС. Обратите внимание на то, что гибридная система MDM является нерекомендуемой. Следует как можно скорее перейти на Intune в Azure.

### <a name="plan-for-change-new-intune-support-experience-for-premier-customers"></a>Планирование изменений. Улучшенная поддержка Intune для клиентов Premier 
Сейчас клиенты Microsoft Premier могут обратиться в службу поддержки Intune на портале Microsoft Premier Online (MPO) (premier.microsoft.com), а также в разделе Intune на портале Azure (portal.azure.com). Начиная с 3 декабря 2018 года, чтобы усовершенствовать работу службы поддержки Premier, мы будем принимать запросы только в разделе Intune на портале Azure.

#### <a name="how-does-this-affect-me"></a>Как это повлияет на работу?
После 3 декабря вы не сможете обращаться в службу поддержки через MPO.  При попытке сделать это вы увидите приглашение перейти в раздел Intune на портале Azure. Это приглашение будет невозможно закрыть. Здесь можно будет создать запрос в службу поддержки Майкрософт, который будет направлен в отдел по работе с Intune для своевременной диагностики и устранения проблемы. Запросы, созданные на портале MPO, не отображаются на портале Azure. Поэтому не следует использовать MPO для этой цели.  

На портале MPO по-прежнему можно создавать запросы на получение поддержки по ConfigMgr (если вы используете гибридное управление мобильными устройствами или совместное управление). Запросы на поддержку Intune следует подавать через портал Azure. Напоминаем, что функция гибридного управления мобильными устройствами больше не поддерживается. Мы рекомендуем перейти на использование Intune в Azure как можно скорее. Дополнительные сведения см. в записи блога [Переход с гибридного управления мобильными устройствами на Intune в Azure](https://aka.ms/hybrid_notification).

Обратите внимание на то, что создавать обращения в службу поддержки на портале Azure могут только пользователи с ролями глобального администратора, администратора службы Intune и администратора поддержки службы.

#### <a name="what-can-i-do-to-prepare-for-this-change"></a>Как можно подготовиться к этому изменению?
- Откажитесь от использования MPO в пользу Intune в Azure, чтобы создавать запросы в службу поддержки Intune и управлять ими.  
- Уведомите свою службу поддержки и при необходимости обновите документацию.
- Если сейчас ваши пользователи без роли глобального администратора или администратора поддержки службы Intune создают запросы через портал MPO, назначьте им роль администратора службы поддержки в Azure Active Directory. Так они смогут создавать запросы в службу поддержки на портале Azure.
- Перейдите по ссылке в разделе "Дополнительные сведения", чтобы получить дополнительную информацию и полезные ссылки.

#### <a name="additional-information"></a>Дополнительные сведения
Дополнительные сведения см. в [записи в блоге службы поддержки Microsoft Intune](https://aka.ms/IntuneSupport_MPO_to_Azure).


### <a name="take-action-please-update-your-android-device-restriction-or-compliance-policy-password-settings-in-intune"></a>Необходимые действия: обновите настройки ограничения устройств Android или параметры пароля в политике соответствия требованиям в Intune
Intune удалит тип пароля "устройство по умолчанию" для устройств Android 4.4 и более поздней версии. Из-за различий в параметрах платформ Android и устройств эта политика часто рассматривается устройством как необязательная. Чтобы устранить неоднозначность в условиях применения этого параметра на устройстве Android, мы удалим этот параметр из пользовательского интерфейса в предстоящем выпуске. 
#### <a name="how-does-this-affect-me"></a>Как это повлияет на работу?
- Если вы хотите требовать пароли на устройствах, рекомендуется вместо использования типа "устройство по умолчанию" изменить профили вашей платформы Android, чтобы четко указать требуемый тип пароля.
- Если вы хотите позволить конечным пользователям решать, нужно ли создавать пароль, выберите "Не настроено". Когда мы удалим этот параметр в пользовательском интерфейсе, если у вас он установлен, вам будет предложено выбрать другое значение, кроме "Устройство по умолчанию" при следующем изменении профиля.
Как можно подготовиться к этому изменению?
Просмотрите параметры пароля в ограничениях устройств и политиках соблюдения требований для Android и Android для бизнеса. Они указаны в разделе системы безопасности для политик соответствия требованиям и в разделе "Пароль устройства" или "Параметры рабочего профиля" для ограничений устройств. В разделе "Дополнительные сведения" приводится ссылка на более подробную информацию и снимки экрана о настройке этих параметров.

### <a name="plan-for-change-intune-moving-to-tls-12"></a>Планирование изменений: переход Intune на TLS 1.2
Начиная с 31 октября 2018 года, Intune будет поддерживать протокол TLS версии 1.2, чтобы обеспечивать лучшее в своем классе шифрование, повысить стандартный уровень защиты нашей службы,а также согласовать возможности с другими службами Майкрософт, такими как Microsoft Office 365. В Office об этом изменении сообщено в MC128929.

31 октября 2018 г. корпоративный портал также начнет поддерживать TLS 1.2.

#### <a name="how-does-this-affect-me"></a>Как это повлияет на работу?
Начиная с 31 октября 2018 года Intune перестанет поддерживать протокол TLS версии 1.0 или 1.1. Все сочетания взаимодействия между клиентом и сервером и браузером и сервером должны использовать протокол TLS версии 1.2, чтобы обеспечить правильное подключение в Intune. Обратите внимание, что это изменение затронет устройства конечных пользователей, которые больше не поддерживаются Intune, но по-прежнему получают политику через Intune, а также не могут использовать протокол TLS версии 1.2. Сюда входят устройства, работающие под управлением Android 4.3 и более ранних версий. Список затронутых устройств и браузеров см. в разделе дополнительной информации ниже.

Если после 31 октября 2018 года у вас возникнет проблема, связанная с использованием старой версии TLS, для ее решения потребуется выполнить обновление до TLS 1.2 или перейти на устройство, поддерживающее TLS 1.2.

#### <a name="what-do-i-need-to-do-to-prepare-for-this-change"></a>Как можно подготовиться к этому изменению?
Рекомендуется заранее удалить зависимости TLS 1.0 и 1.1 в своих средах и по мере возможности отключить протокол TLS 1.0 и 1.1 на уровне операционной системы. Начинайте планировать миграцию на TLS 1.2 уже сегодня. В указанной ниже записи блога службы поддержки приведен список устройств, которые не поддерживаются Intune, но могут по-прежнему получать политику и не смогут взаимодействовать посредством TLS версии 1.2. Может потребоваться уведомить этих конечных пользователей о том, что они потеряют доступ к корпоративным ресурсам.
