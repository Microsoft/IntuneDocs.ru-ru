Имейте в виду, что все политики защиты приложений доступны в колонке **Мобильное приложение** в Intune в разделе **Политики защиты приложений**. Интеграция упрощает управление облачной средой. Дополнительные сведения об изменении политик условного доступа см. в статье [Условный доступ в Azure Active Directory](https://docs.microsoft.com/azure/active-directory/active-directory-conditional-access-azure-portal). #### <a name="passcode-support-for-mam-pin-on-android---1438086---"></a>Поддержка секретного кода для ПИН-кода MAM на Android<!-- 1438086 -->

Администраторы Intune могут задавать условие запуска приложения, требующее ввода секретного кода вместо числового ПИН-кода MAM. Если эта функция настроена, пользователю требуется задать и использовать секретный код при появлении запроса, чтобы получить доступ к приложениям с поддержкой MAM. Секретный код представляет собой числовой ПИН-код, содержащий по меньшей мере один специальный символ или строчные и прописные буквы. Intune поддерживает секретный код так же, как существующий числовой ПИН-код — с возможностью установки минимальной длины и разрешением повторяющихся символов и последовательностей в консоли администрирования. Для этой функции требуется последняя версия корпоративного портала для Android. Функция уже доступна для iOS.

В колонке **Добавить приложение** выберите **Бизнес-приложение**.

### <a name="device-configuration"></a>Конфигурация устройства

####  <a name="device-profile-chart-and-status-list-show-all-devices-in-a-group----1449153-eeready---"></a>Диаграмма профилей устройств и список состояний с отображением всех устройств в группе <!-- 1449153 eeready -->
При настройке профиля устройства (**Конфигурация устройства** > **Профили**) выбирается профиль устройства, например iOS. Затем этот профиль назначается группе, в которую входят устройства iOS и устройства без iOS. В диаграмме показано, что профиль применен к устройствам iOS *и* устройствам без iOS (**Конфигурация устройства** > **Профили** > выберите существующий профиль > **Обзор**). При выборе графической диаграммы на вкладке **Обзор** в разделе **Состояние устройства** вы увидите все устройства в группе, а не только устройства iOS. 

После выхода этого обновления в диаграмме (**Конфигурация устройств** > **Профили** > выбрать существующий профиль > **Обзор**) отображается только количество по конкретному профилю устройств. Например, если профиль конфигурации устройства применяется к устройствам iOS, в диаграмме отображается только количество устройств iOS. При выборе графической диаграммы и открытии раздела **Состояние устройства** указываются только устройства iOS.

Во время внесения этого обновления графическая диаграмма временно отсутствует. 

#### <a name="always-on-vpn-for-windows-10---1333666---"></a>AlwaysOn VPN для Windows 10 <!--1333666 -->

Сейчас [AlwaysOn](https://docs.microsoft.com/windows/security/identity-protection/vpn/vpn-auto-trigger-profile#always-on) можно использовать на устройствах Windows 10 с помощью настраиваемого профиля виртуальной частной сети (VPN), созданного с помощью OMA-URI.

В этом обновлении администраторы могут включать VPN-профили AlwaysOn для Windows 10 непосредственно в Intune на портале Azure. Профили AlwaysOn VPN будут автоматически подключаться в следующих случаях:

- при входе пользователей на устройства;
- при изменении сети устройства;
- при включении экрана устройства после отключения.

#### <a name="new-printer-settings-for-education-profiles----1308900---"></a>Новые параметры принтера для профилей образовательных учреждений <!-- 1308900 -->

Для профилей образовательных учреждений в категории **Принтеры** доступны новые параметры: **Принтеры**, **Принтер по умолчанию**, **Добавить новые принтеры**.

#### <a name="show-caller-id-in-personal-profile---android-for-work---1098984---"></a>Отображение ИД звонящего в личном профиле в программе Android for Work <!--1098984 -->
При использовании личного профиля на устройстве пользователи могут не видеть сведения об ИД звонящего, который является рабочим контактом. 

В этом обновлении в разделе **Android for Work** > **Ограничения устройства** > **Параметры рабочего профиля** появился новый параметр:
- Display work contact caller-id in personal profile (Отображать ИД звонящего, который является рабочим контактом, в личном профиле).

Если этот параметр включен (не настроен), сведения о звонящем, который является рабочим контактом, отображаются в личном профиле. Если этот параметр заблокирован, номер звонящего, который является рабочим контактом, не отображается в личном профиле. 

Область применения: устройства с рабочим профилем Android с ОС Android версии 6.0 или более поздней.

#### <a name="new-windows-defender-credential-guard-settings-added-to-endpoint-protection-settings---1102252-----from-1802-and-1804--"></a>Добавлены новые параметры Credential Guard в Защитнике Windows в раздел параметров Endpoint Protection <!--1102252 --><!--from 1802 and 1804-->

В этом обновлении раздел [Credential Guard в Защитнике Windows](https://docs.microsoft.com/windows/access-protection/credential-guard/credential-guard) (**Конфигурация устройств** > **Профили** > **Endpoint Protection**) включает следующие параметры. 

- **Credential Guard в Защитнике Windows**: активирует Credential Guard с функциями безопасности на основе виртуализации. Эта возможность позволяет защитить учетные данные во время следующей перезагрузки, если включены функции **Уровень безопасности платформы с безопасной загрузкой** и **Безопасность на основе виртуализации**. Возможны следующие значения.
  - **Отключено**: удаленно отключает компонент Credential Guard, если он ранее был включен с параметром **Включено без блокировки**.

  - **Включено с блокировкой UEFI**: блокирует отключение компонента Credential Guard с помощью раздела реестра или групповой политики. Чтобы отключить Credential Guard после использования этого параметра, необходимо отключить групповую политику. Затем следует удалить функции безопасности с каждого компьютера в непосредственном присутствии пользователя. Эти действия очистят конфигурацию, сохраненную в UEFI. Пока сохраняется конфигурация UEFI, компонент Credential Guard включен.

  - **Включено без блокировки**: позволяет удаленно отключать компонент Credential Guard с помощью групповой политики. На устройствах, на которых используется это значение, должна быть установлена по меньшей мере Windows 10 (версии 1511).

При настройке Credential Guard автоматически включаются следующие зависимые технологии. 

  - **Включение безопасности на основе виртуализации (VBS)**: включает функции безопасности на основе виртуализации (VBS) при следующей перезагрузке. Функции безопасности на основе виртуализации используют гипервизор Windows для поддержки служб безопасности и требуют безопасной загрузки.
  - **Безопасная загрузка с прямым доступом к памяти (DMA)**: включает VBS с безопасной загрузкой и прямым доступом к памяти. Средства защиты DMA требуют аппаратной поддержки и включаются только на корректно настроенных устройствах. 

#### <a name="use-a-custom-subject-name-on-scep-certificate----2064190---"></a>Использование пользовательского имени субъекта в сертификате SCEP <!-- 2064190 -->
Вы можете использовать **OnPremisesSamAccountName**, общее имя в пользовательском субъекте профиля сертификата SCEP. Например, можно использовать `CN={OnPremisesSamAccountName})`.

####  <a name="block-camera-and-screen-captures-on-android-for-work----1098977-eeready--"></a>Блокировка снимков экрана и съемки с помощью камеры в Android for Work <!-- 1098977 eeready-->
При настройке ограничений для устройств Android доступны два новых параметра блокировки. 
- Камера. Блокирует доступ ко всем камерам на устройстве.
- Screen capture (Снимок экрана). Блокирует создание снимков экрана, а также запрещает показ содержимого на устройствах отображения, у которых нет безопасного видеовыхода.

Применяется к Android for Work.


### <a name="device-enrollment"></a>Регистрация устройств

#### <a name="new-enrollment-steps-for-users-on-devices-with-macos-high-sierra-10132---1734567---"></a>Новые шаги регистрации для пользователей устройств с macOS High Sierra 10.13.2 и более поздних версий <!--1734567 -->
В macOS High Sierra 10.13.2 представлена концепция регистрации MDM "Утверждено пользователем". Утвержденные регистрации позволяют Intune управлять некоторыми зависящими от безопасности параметрами. Дополнительные сведения см. в сопроводительной документации Apple здесь: https://support.apple.com/HT208019.

Устройства, зарегистрированные с помощью корпоративного портала macOS, считаются неутвержденными пользователем, пока пользователь не откроет системные настройки и не утвердит эти устройства вручную. Поэтому сейчас корпоративный портал macOS направляет пользователей в macOS 10.13.2 и более поздние версии для утверждения регистрации вручную в конце процесса регистрации. После утверждения зарегистрированного устройства пользователем в консоли администрирования Intune будет выведено соответствующее уведомление.



### <a name="device-management"></a>Управление устройствами

#### <a name="advanced-threat-protection-atp-and-intune-are-fully-integrated----eeready-1629303---"></a>Advanced Threat Protection (ATP) и Intune полностью интегрированы <!-- EEready 1629303 -->

[Advanced Threat Protection (ATP)](https://docs.microsoft.com/windows/security/threat-protection/windows-defender-atp/dashboard-windows-defender-advanced-threat-protection) показывает уровень риска устройств Windows 10. Вы можете создать подключение к Microsoft Intune в центре безопасности Защитника Windows (портал ATP). После создания политика соответствия требованиям Intune используется для определения приемлемого уровня угроз. Если уровень угрозы превышен, политика условного доступа Azure Active Directory (AD) может блокировать доступ к различным приложениям в вашей организации.

Эта функция позволяет ATP сканировать файлы, обнаруживать угрозы и сообщать о любых рисках на ваших устройствах Windows 10.

См. сведения в статье [Enable Windows Defender ATP with conditional access in Intune](advanced-threat-protection.md) (Включение ATP в Защитнике Windows с условным доступом в Intune).

#### <a name="support-for-user-less-devices----1637553---"></a>Поддержка устройств без пользователей <!-- 1637553 -->
Intune поддерживает возможность проверять соответствие устройств, не имеющих пользователей, например Microsoft Surface Hub. Политика соответствия может быть назначена конкретным устройствам. Таким образом, устройства, у которых нет соответствующего пользователя, могут быть определены как соответствующие (и несоответствующие).

#### <a name="delete-autopilot-devices----1713650---"></a>Удаление устройств AutoPilot <!-- 1713650 -->
Администраторы Intune могут удалять [устройства AutoPilot](enrollment-autopilot.md#delete-autopilot-devices).

#### <a name="improved-device-deletion-experience---1832333---"></a>Улучшенная процедура удаления устройств <!--1832333 -->
Вам больше не требуется удалять корпоративные данные или сбрасывать устройство до заводских настроек перед [удалением устройства из Intune](devices-wipe.md#delete-devices-from-the-intune-portal).

Чтобы увидеть новый интерфейс, войдите в Intune и выберите **Устройства** > **Все устройства** > имя устройства > **Удалить**.

Если вам по-прежнему требуется подтверждение удаления или снятия с учета, можно использовать стандартные действия из жизненного цикла устройства с выводом параметров **Удалить данные компании** и **Сбросить параметры до заводских настроек** перед нажатием кнопки **Удалить**. 

#### <a name="play-sounds-on-ios-when-in-lost-mode----1947769---"></a>Воспроизведение звуков на устройстве в iOS в режиме пропажи <!-- 1947769 -->
Когда наблюдаемые устройства iOS под управлением MDM находятся в [Режиме пропажи](device-lost-mode.md), вы можете включить [воспроизведение звука](device-locate.md#activate-lost-mode-sound-alert-on-an-ios-device) (**Устройства** > **Все устройства** > выбрать устройство iOS > **Обзор** > **Дополнительно**). Звук будет воспроизводиться, пока устройство не будет выведено из Режима пропажи или пока пользователь не отключит звук на устройстве. Применяется к устройствам iOS 9.3 и более поздних версий.

#### <a name="new-security-enhancements-in-the-intune-service-----1637539---"></a>Новые усовершенствования безопасности в службе Intune <!-- 1637539 -->   

В Intune в Azure представлен переключатель, который пользователи автономной версии Intune могут использовать, чтобы считать устройства без назначенной политики **соответствующими** (функция безопасности отключена) или **несоответствующими** (функция безопасности включена). Это обеспечит доступ к ресурсам только после оценки соответствия устройства политике.

Влияние этой функции зависит от того, имеются ли уже назначенные политики соответствия или нет.

- Для новой или существующей учетной записи, которая еще не назначила устройствам политики соответствия, переключатель автоматически устанавливается в положение **Соответствует**. По умолчанию этот параметр в консоли отключен. На работу конечных пользователей это не влияет.
- Для существующей учетной записи, имеющей устройства с уже назначенной политикой соответствия, переключатель автоматически устанавливается в положение **Не соответствует**. После развертывания мартовского обновления эта функция включена по умолчанию.

Если вы используете политики соответствия требованиям с условным доступом и эта функция включена, устройства, которым не назначена по крайней мере одна политика соответствия требованиям, теперь блокируются функцией условного доступа. Конечные пользователи этих устройств, которым ранее был разрешен доступ к электронной почте, утратят его, если вы не назначите устройствам по крайней мере одну политику соответствия требованиям.   

Обратите внимание, что, несмотря на отображение состояния переключателя по умолчанию в пользовательском интерфейсе сразу же после мартовских обновлений службы Intune, его действие не реализуется немедленно. Все изменения, применяемые к переключателю, не повлияют на соответствие устройства политике, пока мы не назначим вашей учетной записи рабочий переключатель. О завершении этой процедуры мы сообщим вам через Центр сообщений. Это может занять несколько дней после мартовского обновления службы Intune.

#### <a name="configure-the-mac-application-firewall----1690461---"></a>Настройте брандмауэр приложения Mac <!-- 1690461 -->

Вы можете настроить брандмауэр приложения Mac. Используйте эту функцию для управления подключениями на уровне приложений, а не портов. Вам будет проще использовать преимущества брандмауэра и помешать нежелательным приложениям захватить сетевые порты, открытые для разрешенных приложений.

Эта функция находится в разделе **Конфигурация устройства** -> **Создать профиль** -> **macOS** -> **Защита конечной точки**.

После включения параметра брандмауэра можно использовать две стратегии:

- Заблокировать все входящие подключения

   Вы можете заблокировать все входящие подключения для целевых устройств. В этом случае входящие подключения заблокируются для всех приложений.

- Разрешить или блокировать конкретные приложения

   Вы можете разрешить или заблокировать входящие подключения для конкретных приложений. Вы можете включить скрытый режим, чтобы предотвратить ответы на запросы проверки.

####  <a name="detailed-error-codes-and-messages----1376342---"></a>Подробные коды ошибок и сообщения об ошибках <!-- 1376342 -->

В вашей конфигурации устройства можно просмотреть более подробные коды ошибок и сообщения об ошибках. В улучшенных отчетах содержатся параметры, их состояние и подробная информация об устранении неполадок.

##### <a name="more-information"></a>Дополнительные сведения

- Заблокировать все входящие подключения

   Все службы общего доступа (например, общий доступ к файлам или к экрану) будут заблокированы для входящих подключений. Ниже перечислены системные службы, которым будет разрешено принимать входящие подключения:
  - configd: реализует DHCP и другие службы конфигурации сети
  - mDNSResponder: реализует Bonjour
  - racoon: реализует IPSec

    Чтобы использовать службы общего доступа, установите для параметра **Входящие подключения** значение **Не настроено** (не **Блокировать**).

- Скрытый режим

   Включите этот режим, чтобы запретить компьютеру отвечать на запросы проверки. Компьютер будет отвечать на входящие запросы от авторизованных приложений. Неожиданные запросы, например ICMP (проверка связи), будут игнорироваться.

   - **Адрес SIP (протокол инициации сеанса).**

По умолчанию параметры Android for Work совпадают с параметрами устройств Android. Но после изменения параметров Android for Work эта ситуация будет недопустимой.

##### <a name="if-you-have-never-previously-onboarded-android-for-work-enrollment"></a>Если вы ранее никогда не подключали регистрацию Android for Work

Новая платформа Android for Work блокируется в заданных по умолчанию ограничениях по типу устройств. После подключения этой функции будет доступна регистрация устройств в качестве Android for Work. Для этого измените значение по умолчанию или создайте новое ограничение по типу устройств, чтобы заменить значение по умолчанию.

##### <a name="if-you-have-onboarded-android-for-work-enrollment"></a>Если вы подключили регистрацию Android for Work

Если подключение было выполнено ранее, ситуация зависит от выбираемого параметра.

| Параметр | Состояние Android for Work в заданном по умолчанию ограничении по типу устройств | "Заметки" |
| --- | --- | --- |
| **Управлять всеми устройствами в качестве Android** | Блокировано | Все устройства Android необходимо зарегистрировать без Android for Work. |
| **Управлять поддерживаемыми устройствами в качестве Android for Work** | Разрешено | Все устройства, поддерживающие Android for Work, необходимо зарегистрировать в качестве устройств Android for Work. |

#### <a name="windows-defender-application-guard-helps-protect-devices-from-untrusted-websites-as-defined-by-your-organization----958257---"></a>Application Guard в Защитнике Windows защищает устройства от доступа к ненадежным веб-сайтам в соответствии с политикой вашей организации <!-- 958257 -->.

#### <a name="windows-defender-application-control-on-windows-10-enterprise-provides-mode-to-trust-only-authorized-apps----1031096---"></a>Управление приложениями в Защитнике Windows в ОС Windows 10 Корпоративная предоставляет режим для доверия только авторизованным приложениям <!-- 1031096 -->    
В условиях, когда ежедневно создаются тысячи вредоносных файлов, использование антивирусных программ по обнаружению на основе сигнатур может оказаться недостаточным для обеспечения надлежащего уровня защиты от новых атак. С помощью управления приложениями в Защитнике Windows в ОС Windows 10 Корпоративная можно изменить конфигурацию устройства с режима, когда приложения являются доверенными до тех пор, пока не будут заблокированы антивирусной программой или другим решением безопасности, на режим, когда операционная система доверяет только приложениям, авторизованным в вашей организации. Доверие приложениям назначается в управлении приложениями в Защитнике Windows.

В Intune можно настроить политики управления приложениями в режиме "Только аудит" или режиме "Применить". При работе в режиме "Только аудит" приложения не блокируются. В режиме "Только аудит" в журнал записываются все события из журналов событий локальных клиентов. Можно также указать, разрешен ли запуск только для компонентов Windows и приложений Microsoft Store или же возможен запуск дополнительных доверенных приложений, определенных в Intelligent Security Graph.

#### <a name="window-defender-exploit-guard-is-a-new-set-of-intrusion-prevention-capabilities-for-windows-10----1063615---"></a>Exploit Guard в Защитнике Windows — это новый набор функций по предотвращению вторжений для Windows 10 <!-- 1063615 -->   
Exploit Guard в Защитнике Windows содержит пользовательские правила для уменьшения подверженных атакам областей в приложениях, предотвращает угрозы, нацеленные на макросы и скрипты, автоматически блокирует сетевые подключения к ненадежным IP-адресам и защищает данные от программ-вымогателей и неизвестных угроз. Exploit Guard в Защитнике Windows состоит из следующих компонентов:

- **Уменьшение области атаки (ASR)** содержит правила, которые предотвращают угрозы, нацеленные на макросы, скрипты и электронную почту.
- **Управляемый доступ к папкам** автоматически блокирует доступ к содержимому в защищаемых папках.
- **Сетевой фильтр** блокирует исходящие подключения из любого приложения к ненадежному IP-адресу или домену.
- **Защита от эксплойтов** предоставляет ограничения для памяти, потока управления и политики, которые могут использоваться для защиты приложения от эксплойтов.

#### <a name="windows-10-kiosk-mode-device-restrictions----1308872---"></a>Ограничения для применения к устройствам Windows 10 в полноэкранном режиме <!-- 1308872 -->   
Вы можете применить к пользователям устройств Windows 10 ограничения в режиме киоска, чтобы предоставить им доступ только к набору предопределенных приложений.  Для этого создайте профиль ограничения устройств Windows 10 и задайте параметры полноэкранного режима.

Полноэкранный режим поддерживает два режима: **одно приложение** (позволяет пользователю запускать только одно приложение) или **несколько приложений** (разрешает доступ к набору приложений).  Вы определяете учетную запись и имя устройства, которое определяет поддерживаемые приложения.  Когда пользователь выполняет вход, ему будут доступны только определенные приложения.  Дополнительные сведения см. в статье [AssignedAccess CSP](https://docs.microsoft.com/windows/client-management/mdm/assignedaccess-csp). 

Для полноэкранного режима необходимо выполнить следующие условия:

- Устройство должно быть [подготовлено надлежащим образом](https://docs.microsoft.com/windows/configuration/set-up-a-kiosk-for-windows-10-for-desktop-editions).

### <a name="plan-for-change-new-windows-10-setting-for-kiosk-configuration-in-intune----1560072---"></a>Планирование изменений: новый параметр Windows 10 для конфигурации киоска в Intune <!-- 1560072 -->
Мы работаем над изменениями способа и места настройки рабочих столов Windows 10 1709 и более поздних версий (RS3 и более поздних версий) на портале Intune Azure.

#### <a name="how-does-this-affect-me"></a>Как это повлияет на работу? 
По нашим сведениям, вы выбираете "Windows 10" > "Ограничения устройств" и используете параметр "Киоск (предварительная версия)". В мае этот параметр пользовательского интерфейса будет переименован в "Киоск (устаревший)", чтобы указать, что его не рекомендуется использовать. В качестве альтернативы в мае мы выпускаем новый профиль конфигурации устройства: "Windows 10" > "Киоск", содержащий параметры для настройки киосков в Windows 10 RS4 и более поздних версиях.

Это изменение затронет клиентов как автономного решения Intune, так и гибридного решения (Intune с Configuration Manager). Интеграция упростит управление облачной средой. Для управления группами, политиками, приложениями и мобильными устройствами будет использоваться одна колонка в Azure — колонка Intune.

#### <a name="what-do-i-need-to-do-to-prepare-for-this-change"></a>Как можно подготовиться к этому изменению?
Добавьте в избранное колонку Intune вместо колонки службы "Защита приложений Intune" и ознакомьтесь с рабочим процессом политики защиты приложений в колонке "Мобильное приложение" в Intune. В течение некоторого времени будет происходить перенаправление, после чего колонка "Защита приложений Intune" будет удалена. Помните, что все политики защиты приложений уже перенесены в Intune и вы можете изменять любые политики условного доступа, следуя документации: [https://aka.ms/azuread_ca](https://aka.ms/azuread_ca).

Более года назад мы объявили о выпуске [общедоступной предварительной версии Intune в Azure](https://cloudblogs.microsoft.com/enterprisemobility/2016/12/07/public-preview-of-intune-on-azure/), а шесть месяцев назад представили [общедоступную версию нового интерфейса администрирования](https://cloudblogs.microsoft.com/enterprisemobility/2017/06/08/the-new-intune-and-conditional-access-admin-consoles-are-ga/) для Intune. Начиная со 31 августа 2018 г. функции управления мобильными устройствами (MDM) в классической консоли Silverlight будут отключены для клиентов, использующих изолированную службу Intune. Вместо этого вы можете использовать [Intune в Azure](https://aka.ms/Intune_on_Azure) для управления мобильными устройствами. Если вы все еще используете классическую консоль для управления мобильными устройствами, рекомендуем перейти на Intune в Azure. Это изменение не должно повлиять на работу конечных пользователей. Управление обычными компьютерами будет по-прежнему осуществляться в консоли Silverlight. Дополнительные сведения об этом изменении и его последствиях см. [здесь](https://aka.ms/Intune_on_Azure_mdm).

### <a name="direct-access-to-apple-enrollment-scenarios---951869--"></a>Прямой доступ к сценариям регистрации Apple <!--951869-->
Для учетных записей Intune, созданных после января 2017 г., включен прямой доступ к сценариям регистрации Apple с помощью рабочей нагрузки регистрации устройств на портале Azure. Ранее доступ к предварительной версии регистрации Apple можно было получить, только перейдя по ссылкам на классическом портале Intune. Учетные записи Intune, созданные до января 2017 г., требуется перенести (единожды), перед тем как эти возможности станут доступны в Azure. Расписание переноса еще не объявлено, но мы предоставим сведения о нем в кратчайшие сроки. Если вам не удается получить доступ к порталу Azure, используя имеющуюся учетную запись, мы настоятельно рекомендуем создать пробную учетную запись, чтобы протестировать новые возможности.

## <a name="whats-coming"></a>Предстоящие изменения

### <a name="local-device-security-option-settings----1251887---"></a>Локальные параметры безопасности устройства <!-- 1251887 -->
Вы сможете включать параметры безопасности на устройствах с Windows 10 с помощью новых локальных параметров безопасности устройства. Эти параметры можно найти в категории Endpoint Protection при создании политики конфигурации устройства с Windows 10.

### <a name="new-user-experience-update-for-the-company-portal-website---2000968--"></a>Обновление пользовательского интерфейса на веб-сайте корпоративного портала <!--2000968-->

В апреле мы представляем новый интерфейс веб-сайта корпоративного портала с упрощенными рабочими процессами и улучшенными специальными возможностями. В нем реализованы такие ориентированные на клиента усовершенствования, как общий доступ к приложениям, а также повышена общая производительность, что делает работу с порталом удобнее.
Основываясь на отзывах клиентов, мы добавили ряд новых возможностей, которые значительно расширяют существующую функциональность и повышают удобство работы:

-   улучшения пользовательского интерфейса веб-сайта;
-   возможность делиться прямыми ссылками на приложения;
- повышение производительности для больших каталогов приложений.

Для подготовки к этому изменению никакие действия выполнять не нужно. Мы сообщим вам, когда обновленный веб-сайт корпоративного портала станет доступен. Но вам может потребоваться обновить снимки экрана в документации для пользователей. Обратите внимание на то, что вам также может потребоваться обновить документацию по приложению "Корпоративный портал" для iOS, так как раздел **Приложения** в нем зависит от веб-сайта. Изображение с примером см. в описании [новых возможностей в пользовательском интерфейсе приложения](whats-new-app-ui.md).

### <a name="apple-to-require-updates-for-application-transport-security---748318--"></a>Требуемые обновления Application Transport Security от Apple <!--748318-->
Компания Apple объявила, что к Application Transport Security (ATS) будут применяться определенные требования. ATS позволяет обеспечить высокий уровень безопасности при передаче данных в приложениях по протоколу HTTPS. Это изменение затрагивает клиентов Intune, которые используют приложения корпоративного портала для iOS. Подробные сведения будут приведены в нашем [блоге поддержки Intune](https://aka.ms/compportalats).
